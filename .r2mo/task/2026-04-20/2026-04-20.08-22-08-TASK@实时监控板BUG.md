## Changes
- 2026-04-17: [AI Agent] Fixed all monitor board bugs per task requirements
  - **Live sessions only**: Removed static file-based session scanning. `scanLiveSessions()` now only detects processes actually running in terminals (WezTerm, iTerm2, Terminal.app) by walking process trees via `getAllChildPids()` up to 6 levels deep.
  - **Working/Ready status**: Uses `proc_pidinfo(PROC_PIDTASKINFO)` to read CPU ticks. Processes with active CPU usage > 0.5 → Working (green), idle → Ready (blue).
  - **Real-time refresh**: Timer interval changed from 15s to 5s. Auto-refreshes monitor tab when open.
  - **AI tool icons**: Each session row shows the AI tool's icon (Claude, Codex, OpenCode, Cursor, Trae) from `:/ai-tools/` resource via `SessionScanner::toolIconPath()`.
  - **Project list from vaults**: `collectAllProjectPaths()` scans all added vaults via `R2moScanner`, not hardcoded.
  - **Flat session model**: Replaced `ToolSessionSummary` intermediate layer with direct `ProjectMonitorData::sessions` list for simpler tree rendering.

- 2026-04-17: [AI Agent] UI refinement + status fix
  - **Flat table layout**: 6 columns (Project, Terminal, AI Tool, Session, Status, Action) evenly distributed with `Stretch`.
  - **Project in first row**: No nesting, no count badges. Project name directly in col 0.
  - **AI tool icons**: Claude/Codex/OpenCode/Cursor/Trae icons from `:/ai-tools/` shown in AI Tool column.
  - **Session & Status split**: Session shows PID, Status shows colored dot (green=Working, blue=Ready).
  - **Action column**: "Detail" button that activates the terminal window via AppleScript (`osascript`) then opens session detail dialog.
  - **Status detection fix**: Uses CPU tick delta (`pti_total_user + pti_total_system`) between scans. Delta > 100k ticks = Working, otherwise Ready. 30s grace period to avoid flicker.
  - **5s auto-refresh**: Real-time polling for live status updates.

  - **No full-tab flicker**: Monitor refresh no longer swaps the whole tab out for a loading overlay on each poll; refreshed content is replaced in-place to avoid visible flashing.

  - **Consecutive activity rule**: `Working` now prefers two consecutive active samples over single short CPU spikes, reducing false positives on idle Codex processes.
  - **Short live hold**: A session falls back to `Ready` quickly unless activity remains continuous.

- 2026-04-17: [AI Agent] Active-session tightening + responsive width follow-up
  - **Active vs ready refinement**: Session status now combines process running-state (`SRUN`) with normalized CPU delta-per-second, so only truly active shells/processes are promoted to `Working` while most idle Codex/OpenCode sessions remain `Ready`.
  - **Stable session row identity**: Status-only refresh now keys rows by `project + tool + stable session id`, avoiding unnecessary table rebuilds when the representative child PID changes inside the same session.

  - **AI Tool width widened again**: The `AI Tool` column budget was expanded once more so `OpenCode` remains visible in narrower monitor layouts.

- 2026-04-17: [AI Agent] CLI transcript-driven execution detection
  - **Direct CLI activity signal**: `Working` now prefers local session artifact growth over CPU heuristics. For Codex and Claude, the monitor binds each terminal session to its real transcript file and treats recent append/mtime growth as active model execution.
  - **Codex session binding**: When `codex resume <sessionId>` appears in the command line, the monitor resolves that exact `rollout-*.jsonl` transcript instead of guessing from generic CPU activity.
  - **OpenCode fallback artifact**: OpenCode still lacks an equivalent local transcript in this environment, so its workspace `.dat` updates are used as the nearest available activity signal before falling back to process heuristics.
  - **Stable refresh ordering**: Session rows are no longer sorted by current status, so a `Ready`/`Working` flip does not reshuffle the table and break status-cell-only refresh.

- 2026-04-17: [AI Agent] Transcript event-semantic status algorithm
  - **Codex semantic status**: `Working/Ready` is now inferred from transcript tail events, with `task_started`, tool/function call phases, and reasoning phases treated as active, while `task_complete` is treated as terminal `Ready`.
  - **Claude semantic status**: Claude transcript tails now treat `tool_use`, `PreToolUse`, and `PostToolUse` as active execution phases, while `Stop`, `stop_hook_summary`, and `end_turn` are treated as completed/ready phases.
  - **OpenCode paused handling**: OpenCode workspace state now explicitly maps `paused` sessions to `Ready` instead of leaving them to CPU heuristics.
  - **CPU heuristic demoted**: Process-state and CPU delta logic now only act as a fallback when no reliable transcript/workspace semantic signal can be extracted.

- 2026-04-17: [AI Agent] Final active/ready correction with session-bound semantics
  - **Real session id plumbing**: `determineStatus(...)` now receives the resolved `sessionId` directly, fixing the previous bug where `OpenCode` status lookup used the artifact filename instead of the real `ses_...` id.
  - **Last-event status inference**: `Codex` and `Claude` no longer scan the whole transcript tail for any historical keyword; they parse the last valid JSON event/object and infer `Working/Ready` from the current terminal phase only.
  - **More conservative OpenCode rule**: `OpenCode` now prefers explicit `turn-complete` / `error` / session-specific `paused` signals and otherwise falls back to `Ready`, avoiding broad false-positive `Working` states caused by generic workspace blobs.
  - **Initial sample no longer flashes Working**: CPU fallback now starts from `Ready` on the first sample, so newly discovered idle sessions do not briefly turn green before enough runtime evidence exists.

- 2026-04-18: [AI Agent] Claude idle-session false positive correction
  - **Root cause fixed**: Some idle `claude --continue` sessions were still shown as `Working` because the transcript tail ended with `attachment` stop hooks or `system/away_summary`, which the monitor did not yet classify as terminal-ready events.
  - **Claude tail coverage expanded**: `attachment` records with `hookEvent=Stop`, `hook_success`, and `async_hook_response`, plus `system/away_summary` and `permission-mode`, are now treated as `Ready`.
  - **Working only on real active edges**: Claude remains `Working` only when the last tail object is a real active edge such as `PreToolUse`, `PostToolUse`, assistant `stop_reason=tool_use`, or explicit `tool_use`.
